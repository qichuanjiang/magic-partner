# 需求规格：image-upload-llm-analysis

## 1. 背景

当前项目仅提供基础页面与示例 API（健康检查、问候语），缺少真实业务链路示例。
本需求新增「图片上传 -> 后端调用大模型分析 -> 前端展示结果」闭环，作为后续多模态能力的基础能力。

## 2. 目标

- 用户可在前端上传单张图片并触发分析。
- 后端完成图片校验后调用大模型图片理解能力，返回结构化分析结果。
- 前端可稳定展示分析摘要、标签和置信度信息。

## 3. 非目标

- 不实现用户系统、权限分级和多租户隔离。
- 不实现对象存储（OSS/S3）长期持久化，V1 仅做临时处理。
- 不实现异步任务队列与任务中心（V1 采用同步调用）。
- 不实现批量上传、多图对比、历史记录检索。

## 4. 用户场景

- 场景 A：运营人员上传一张商品图，系统返回图片内容摘要和分类标签，便于快速录入。
- 场景 B：客服上传用户截图，系统返回关键元素识别结果，辅助人工判断。
- 场景 C：当上传非法文件或分析失败时，用户可收到明确错误提示并重试。

## 5. 功能需求

- FR-1：前端提供图片选择与上传入口，支持 `jpg/jpeg/png/webp`。
- FR-2：前端在提交前完成基础校验（类型、大小），并显示校验失败原因。
- FR-3：后端接口接收 `multipart/form-data`，字段名为 `image`。
- FR-4：后端对文件类型、大小进行二次校验，防止绕过前端校验。
- FR-5：后端调用大模型图片分析能力，生成统一响应结构。
- FR-6：前端展示分析结果：`summary`、`tags`、`confidence`、`request_id`。
- FR-7：接口错误时返回可读错误信息；前端按错误码分场景提示。
- FR-8：后端记录核心日志（请求 ID、耗时、结果状态、错误码）。

## 6. API 契约

- 接口路径：`/api/images/analyze`
- 请求方法：`POST`
- Content-Type：`multipart/form-data`
- 请求参数：
  - `image`（必填，文件）
  - `prompt`（可选，字符串；用于覆盖默认分析指令）
- 约束：
  - 图片格式：`image/jpeg`、`image/png`、`image/webp`
  - 单文件大小：不超过 `5MB`

- 成功响应（200）：

```json
{
  "request_id": "req_20260306_001",
  "summary": "一张室内拍摄的咖啡杯图片，背景有笔记本电脑。",
  "tags": [
    { "name": "coffee", "confidence": 0.98 },
    { "name": "laptop", "confidence": 0.91 },
    { "name": "indoor", "confidence": 0.88 }
  ],
  "model": "gpt-4.1-mini",
  "latency_ms": 1320,
  "created_at": 1772755200000
}
```

- 失败响应（示例）：

```json
{
  "request_id": "req_20260306_001",
  "error": {
    "code": "UNSUPPORTED_IMAGE_TYPE",
    "message": "仅支持 jpg/jpeg/png/webp 图片"
  }
}
```

- 错误码：
  - `INVALID_IMAGE_SIZE`（413）：文件超出大小限制
  - `UNSUPPORTED_IMAGE_TYPE`（400）：文件类型不支持
  - `MODEL_TIMEOUT`（504）：模型调用超时
  - `MODEL_UPSTREAM_ERROR`（502）：模型服务异常
  - `INVALID_REQUEST`（422）：请求参数缺失/格式错误

## 7. 数据模型 / 持久化

- V1 不新增数据库表。
- 仅在内存中处理上传内容并传递给模型 SDK。
- 日志中保留：`request_id`、文件元信息（不记录原始图片内容）、错误码、耗时。

## 8. 安全与权限

- 鉴权：V1 与现有公开接口一致，不新增鉴权逻辑（仅限内网/受控环境）。
- 输入安全：严格校验 MIME 与大小，拒绝非图片内容。
- 隐私保护：
  - 不在日志中记录图片原始二进制。
  - 不在日志中记录用户敏感文本。
- 配置安全：模型 API Key 通过环境变量注入，不写入代码仓库。

## 9. 风险与缓解

- 风险 1：模型响应耗时高，导致接口超时。
  - 缓解：设置超时（如 15s）并返回 `MODEL_TIMEOUT`；前端提示重试。
- 风险 2：模型返回非结构化文本，前端难以渲染。
  - 缓解：后端统一解析为固定 JSON 结构并做字段兜底。
- 风险 3：大图或恶意文件导致资源占用过高。
  - 缓解：前后端双重大小限制 + 后端 MIME 校验 + 请求体限制。
- 风险 4：上游模型波动导致成功率下降。
  - 缓解：增加错误码分层和日志告警，必要时快速降级为“暂不可用”。

## 10. 发布与回滚

- 发布策略：
  - 后端先发布接口并通过手工验证。
  - 前端上线上传入口并默认开启（如需可加 feature flag）。
- 回滚策略：
  - 回滚后端到上一个稳定镜像/标签，恢复无该接口状态。
  - 前端回滚到上一版本，隐藏上传入口。
  - 若仅模型调用异常，可保留页面并在后端返回统一降级提示。

## 11. 待确认问题

- Q1：模型供应商与模型版本最终选型（示例中为 `gpt-4.1-mini`）。
- Q2：是否需要内容安全审核（涉黄/涉政/涉暴）并在结果中给出风险标记。
- Q3：V1 是否必须持久化分析结果，还是仅前端即时展示即可。
- Q4：生产环境是否需要接口鉴权与限流（IP/QPS）。
