# 功能规格说明（`spec.md`）

## 基本信息

- **功能名称**：人物声音采集与语音聊天
- **规格目录**：`specs/persona-voice-chat/`
- **创建日期**：2026-03-12
- **状态**：`draft`
- **输入来源**：用户对“数字分身 / 娱乐陪聊”原型的需求说明

## 1. 问题与目标

### 1.1 问题背景

当前项目缺少一条完整的“创建人物 -> 采集人物声音 -> 配置人物设定 -> 通过文字或语音与人物聊天 -> 收到人物语音回复”的业务闭环。用户希望先做一个本地原型，验证人物管理、声音采集、声音克隆与语音聊天链路是否可用，再决定是否继续扩展为更完整的产品。

### 1.2 目标

- **业务目标**：提供一个可演示、可验证的数字人物语音聊天原型。
- **用户目标**：用户可以创建人物、上传或录制声音样本、编辑人物设定，并与已就绪人物进行文字或语音聊天。
- **交付目标**：首版完成从人物创建到人物语音回复的完整闭环，并支持保留最近 10 轮聊天记录，以及一键重置人物状态。

### 1.3 非目标

明确本次不做什么，避免范围蔓延。

- 不包含：登录、注册、用户隔离、权限控制、支付、推荐、审核后台
- 不扩展：视频生成、形象驱动、实时视频通话、复杂风控、内容安全体系
- 不重构：数据库、对象存储、消息队列、异步任务等超出原型范围的基础设施

## 2. 用户角色

列出本功能涉及的主要角色，只保留真实会使用该功能的人或系统。

| 角色 | 描述 | 关键诉求 |
| --- | --- | --- |
| 访问者 | 在网页中创建和使用人物进行语音聊天的用户 | 快速创建人物、顺利完成声音采集和聊天 |
| 系统 | 负责管理人物、样本、聊天与语音生成的应用 | 稳定返回明确状态和错误结果 |

## 3. 用户故事与验收场景

要求：

- 按优先级排序，`P1` 必须能单独构成最小可交付价值
- 每个故事都必须“独立可测试”
- 每个故事都必须写清楚验收条件

### 用户故事 1 - 创建并准备可聊天人物（Priority: P1）

**故事描述**  
作为访问者，我想要创建一个人物、录制或上传声音样本，并看到人物是否已就绪，从而后续可以在聊天页中选择该人物。

**为什么是 P1**  

没有人物创建、声音样本采集和就绪判断，就不存在后续聊天入口。这是整条业务链路的起点，也是最小可交付价值。

**独立测试方式**  

只实现这一条时，用户应能成功创建人物、为该人物上传合法样本、看到样本统计信息和“已就绪/未就绪”状态，并能删除样本或人物。

**验收场景**

1. **Given** 用户输入合法且未重复的人物名，**When** 创建人物，**Then** 系统成功保存人物并在列表中展示该人物
2. **Given** 人物已有合法声音样本且满足就绪条件，**When** 用户查看人物列表，**Then** 系统展示该人物为“已就绪”
3. **Given** 任一上传文件格式或时长不合法，**When** 用户提交一批样本上传，**Then** 系统整体拒绝该批次且不保存任何新文件

### 用户故事 2 - 调整人物设定并维护人物状态（Priority: P2）

**故事描述**  
作为访问者，我想要编辑人物设定，并在需要时重置人物设定与聊天记录，从而重新开始调教该人物。

**为什么是 P2**

人物设定决定聊天风格和输出内容，是体验质量的重要组成部分；重置功能则用于快速回到初始状态，便于反复调教和演示。

**独立测试方式**

只实现这一条时，用户应能修改人物设定、保存后再次打开仍可看到最新设定，并能通过“重置人物状态”清空人物设定和聊天记录。

**验收场景**

1. **Given** 用户修改人物设定字段，**When** 保存设定，**Then** 系统持久化保存最新设定
2. **Given** 某人物已有设定和聊天记录，**When** 用户执行重置人物状态，**Then** 系统清空该人物设定和聊天记录

### 用户故事 3 - 与人物进行文字或语音聊天（Priority: P3）

**故事描述**  
作为访问者，我想要选择一个已就绪人物，通过文字或语音发起对话，并收到该人物的文本与语音回复，从而体验数字人物聊天效果。

**为什么是 P3**

聊天是最终体验目标，但它依赖人物创建、声音样本采集和人物设定等前置能力，因此排在后面。

**独立测试方式**

只实现这一条时，用户应能选择一个已就绪人物，输入文本或语音，系统生成符合人物设定的文本回复，并尝试使用该人物音色输出语音；若音色生成失败，应明确提示人物当前无法说话。

**验收场景**

1. **Given** 用户选择一个已就绪人物并输入文字，**When** 发送消息，**Then** 系统返回文本回复和可播放的语音回复
2. **Given** 用户录制语音作为输入，**When** 识别成功并发送，**Then** 系统展示识别文本并返回该人物的语音回复
3. **Given** 人物音色生成当前不可用，**When** 系统尝试合成回复语音，**Then** 系统明确提示“当前人物说不出话了”这一类失败信息，且不以默认 TTS 冒充该人物发声

## 4. 边界与异常场景

至少覆盖以下类型：

- 空输入、非法输入、边界输入
- 外部依赖失败
- 权限/状态不满足
- 重复提交或幂等问题
- 前后端数据不一致时的处理

示例占位：

- 当人物名为空、超长、包含非法字符或大小写意义下重复时，系统应阻止创建或重命名并返回明确错误
- 当任一上传音频格式不支持、时长过短、时长过长或超过单人物样本上限时，系统应整体拒绝该批上传并回滚已完成写入
- 当用户尝试选择未就绪人物发起聊天时，系统应拒绝请求并返回人物未就绪错误
- 当语音识别失败时，系统应保留用户继续编辑文本并手动发送的能力
- 当回复生成失败、语音克隆失败、文件写入失败或删除目标不存在时，系统应返回明确错误，不允许隐式失败

## 5. 功能需求

要求：

- 只写“系统必须做什么”，不要写实现细节
- 每条需求都要可测试、可追踪
- 编号连续，便于后续任务引用

### 5.1 功能性需求

- **FR-001**：系统必须提供首页或导航页，并提供进入人物管理页和聊天页的入口
- **FR-002**：系统必须提供人物管理页，支持创建、查看、重命名和删除人物
- **FR-003**：每个人物必须有稳定的内部标识，不得仅依赖目录名表达业务身份
- **FR-004**：每个人物必须包含人物名、创建时间、更新时间和可编辑的人物设定字段，设定字段至少包括 `description`、`speaking_style`、`catchphrases`、`persona_notes`
- **FR-005**：人物名必须为必填，长度限制为 `1-30` 个字符，仅允许中文、英文、数字、空格、`-`、`_`
- **FR-006**：系统比较人物名时必须大小写不敏感；若名称重复，系统必须拒绝创建或重命名并返回明确错误
- **FR-007**：系统必须支持浏览器直接录音和本地音频文件上传，并支持一次选择多个文件
- **FR-008**：服务端必须统一将合法声音样本转存为 `wav`
- **FR-009**：系统必须支持的上传格式至少包括 `mp3`、`wav`、`m4a`、`webm`
- **FR-010**：单段音频时长必须限制在 `3-60` 秒之间；前端提交前必须校验，后端必须再次执行同等校验
- **FR-011**：单个人物最多保存 `50` 个音频文件；达到上限后系统必须拒绝继续上传
- **FR-012**：任一上传文件不合法时，该批次上传必须整体失败，且不得保存任何本批次新文件
- **FR-013**：系统必须为每个人物保存样本统计信息，至少包括样本数量、总时长、最近上传时间和就绪状态
- **FR-014**：人物满足“至少 `3` 段录音且总录音时长至少 `30` 秒”后，系统才可将其标记为“已就绪”
- **FR-015**：未就绪人物不得在聊天页中被选择，系统必须展示未就绪原因或进度信息
- **FR-016**：系统必须支持试听人物样本，默认播放最新上传的一条样本；每条样本必须支持删除
- **FR-017**：删除样本后，人物样本统计和就绪状态必须同步更新
- **FR-018**：系统必须支持编辑人物设定字段，且设定字段允许为空
- **FR-019**：人物设定变更后，后续回复必须以当前设定为准
- **FR-020**：系统必须提供“重置人物状态”功能，至少清空该人物的设定字段、聊天记录和共享上下文
- **FR-021**：聊天页必须支持选择已就绪人物，并支持搜索与排序；排序至少包括最近使用、最近更新和名称排序
- **FR-022**：聊天页必须支持文字输入和语音输入；语音输入必须先经过语音识别
- **FR-023**：语音识别成功后，系统必须展示识别文本；识别失败时，系统必须允许用户编辑文本并手动发送
- **FR-024**：聊天请求发起时，必须明确包含当前选择的人物
- **FR-025**：后端必须基于当前人物设定、当前人物共享上下文和用户当前输入生成回复
- **FR-026**：回复文本必须避免出现“作为 AI”之类免责声明式表述
- **FR-027**：回复输出必须包含可展示的文本内容
- **FR-028**：回复输出必须包含可播放的语音结果，且该语音结果应来自人物真实音色克隆能力，而不是固定默认 TTS 冒充
- **FR-029**：当人物音色生成不可用或失败时，系统必须明确告知用户类似“当前人物说不出话了”的提示
- **FR-030**：系统允许在回复过程中展示处理中状态，至少支持“识别中”“思考中”“合成中”
- **FR-031**：聊天上下文和聊天记录必须按人物维度隔离保存，不得因页面刷新产生隐式重置
- **FR-032**：每个人物只保留最近 `10` 轮上下文
- **FR-033**：每个人物只保留最近 `10` 轮聊天记录
- **FR-034**：`1` 轮定义为“用户一条消息 + AI 一条消息”
- **FR-035**：聊天记录至少应包含发言方、文本内容、语音播放信息、时间戳和异常状态
- **FR-036**：AI 回复语音必须支持重复播放；历史消息在保留窗口内必须可再次播放
- **FR-037**：超出保留窗口的较老消息可以仅保留文本，或被整体清理，但系统行为必须稳定一致
- **FR-038**：系统必须支持删除人物；删除前前端必须进行二次确认
- **FR-039**：删除人物时，系统必须同步删除原始声音样本、人物元数据、聊天记录、生成语音和上下文数据
- **FR-040**：删除不存在的人物或资源时，系统必须返回明确错误

### 5.2 非功能性需求

- **NFR-001**：错误必须显式处理，不允许隐式失败
- **NFR-002**：新增 API 必须定义成功响应与关键错误响应
- **NFR-003**：新增前端交互必须有明确加载态、成功态、失败态
- **NFR-004**：实现必须保持在现有 FastAPI + Vue 3 + Vite 技术边界内，除非本规格明确说明例外
- **NFR-005**：后端必须统一封装模型能力调用，前端不得直接调用第三方 ASR、LLM、TTS 或声音克隆服务
- **NFR-006**：路由层只负责 HTTP 协议适配，不承担音频转码、存储编排和业务判断细节
- **NFR-007**：服务层必须显式区分人物管理、声音样本管理、聊天上下文管理、对话编排和语音输出
- **NFR-008**：错误响应必须区分用户可修复错误与系统内部错误，并为关键错误提供稳定错误码或等价稳定标识
- **NFR-009**：批量上传或批量写入失败时，系统必须回滚已完成的部分写入
- **NFR-010**：首版默认采用“文件存磁盘 + 元数据存 JSON”，但代码边界必须允许未来迁移到对象存储和数据库
