# 人物声音采集与语音聊天

## 功能概述

提供一个面向“数字分身 / 娱乐陪聊”场景的人物管理与语音聊天功能。用户可在网页中创建人物，录制或上传该人物的声音样本，编辑人物设定，并在聊天页选择已就绪人物，通过文字或语音输入发起对话。系统根据人物设定、共享上下文和用户输入生成回复，并以语音形式返回，同时展示对应文本。

首版聚焦本地原型验证，默认采用“文件存磁盘 + 人物元数据存 JSON”的方式，不引入登录、数据库、对象存储、风控审核和复杂权限系统。

## 宪法对齐检查

- 是否以规格先行：是。本文件先明确人物、样本、聊天、上下文和删除边界，再进入实现阶段。
- 是否遵循最小必要功能：是。首版仅覆盖人物管理、声音样本采集、人物设定编辑、文字/语音输入聊天、语音回复与历史保留，不包含登录、多用户隔离、支付、推荐、审核后台等扩展功能。
- 是否显式处理错误：是。人物名非法、人物重名、音频格式非法、时长不合法、样本不足、识别失败、生成失败、TTS 失败、删除目标不存在等场景均要求显式处理。
- 是否避免不必要抽象：是。首版允许使用本地文件系统和 JSON 元数据，但要求在接口和服务边界上为后续接入对象存储和数据库留出空间。
- 是否满足先测后实现：是。后续实现必须遵循先写失败测试、再实现、再重构的顺序。

## 目标与非目标

### 目标

1. 提供人物管理页，支持创建、查看、重命名、删除人物。
2. 提供声音采集入口，支持浏览器录音和本地音频文件上传。
3. 为每个人物保存可编辑的人物设定。
4. 提供聊天页，支持选择人物后以文字或语音输入发起对话。
5. 后端根据人物设定、人物共享上下文和用户输入生成回复。
6. 系统以语音形式返回回复，并同步展示回复文本。
7. 支持按人物清除上下文，使后续回复不再引用旧对话。

### 非目标

1. 不实现登录、注册、用户隔离和权限系统。
2. 不限制访问者上传他人声音。
3. 不实现审核、授权确认、内容安全策略和滥用防护。
4. 不实现数据库、对象存储、消息队列或异步任务系统。
5. 不实现视频生成、虚拟人形象驱动或实时视频通话。
6. 不保证首版达到真实商业级“声音克隆”效果，只要求具备完整业务链路和可验证行为。

## 用户故事

### 1. 人物创建

作为访问者，我希望创建一个人物，并为其录制或上传声音样本，以便后续在聊天中选择该人物进行语音回复。

### 2. 人物调教

作为访问者，我希望编辑人物的自由描述、说话风格、口头禅和补充设定，以便系统生成的回复更接近我期望的人设。

### 3. 文字聊天

作为访问者，我希望在聊天页选择一个已就绪人物并输入文字，系统返回该人物风格的语音回复，同时展示回复文本。

### 4. 语音聊天

作为访问者，我希望用语音输入消息，系统先识别文本，再按人物设定生成语音回复。

### 5. 上下文控制

作为访问者，我希望一个人物的聊天保留一定上下文，也希望在需要时一键清空，让该人物从新的状态重新开始聊天。

## 页面与导航

### 1. 首页 / 导航页

1. 系统必须提供首页或导航页。
2. 首页必须提供进入人物管理页和聊天页的入口。
3. 首页应说明当前产品定位为人物声音采集与语音聊天原型。

### 2. 人物管理页

1. 系统必须提供人物列表。
2. 系统必须支持创建人物。
3. 系统必须支持编辑人物设定。
4. 系统必须支持录音和上传声音样本。
5. 系统必须支持试听人物样本。
6. 系统必须支持重命名人物。
7. 系统必须支持删除人物。
8. 系统必须明确展示人物是否“已就绪”。

### 3. 聊天页

1. 系统必须提供人物选择器。
2. 人物选择器只能选择“已就绪”人物。
3. 人物选择器必须支持搜索。
4. 人物列表必须支持排序，至少包括：
   - 最近使用
   - 最近更新
   - 名称排序
5. 聊天页必须支持文字输入。
6. 聊天页必须支持语音输入。
7. 聊天页必须展示消息历史。
8. 聊天页必须提供清除上下文功能。

## 功能性需求

### 1. 人物模型

1. 每个人物必须有稳定的内部标识，不得直接以目录名作为唯一业务实体。
2. 每个人物必须包含人物名。
3. 每个人物必须支持以下设定字段：
   - `description`
   - `speaking_style`
   - `catchphrases`
   - `persona_notes`
4. 每个人物必须包含创建时间和更新时间。
5. 每个人物必须包含样本统计信息，至少包括：
   - 样本数量
   - 总时长
   - 最近上传时间
   - 就绪状态
6. 每个人物必须可被重命名。
7. 人物重命名后，相关目录、元数据和关联资源必须同步更新。

### 2. 人物命名规则

1. 人物名为必填项。
2. 人物名长度限制为 `1-30` 个字符。
3. 人物名仅允许：
   - 中文
   - 英文
   - 数字
   - 空格
   - `-`
   - `_`
4. 人物名比较时必须大小写不敏感。
5. 若人物名重复，系统必须拒绝创建或重命名，并返回明确错误。
6. 若人物名不合法，系统必须阻止保存，并返回明确错误。

### 3. 声音样本采集

1. 系统必须支持浏览器直接录音。
2. 系统必须支持上传本地音频文件。
3. 上传时必须支持一次选择多个文件。
4. 服务端必须统一将声音样本转存为 `wav`。
5. 支持的上传格式至少包括：
   - `mp3`
   - `wav`
   - `m4a`
   - `webm`
6. 单段音频时长不得短于 `3` 秒。
7. 单段音频时长不得长于 `60` 秒。
8. 单个人物最多保存 `50` 个音频文件。
9. 达到上限后，系统不得继续接收新文件，必须提示用户先删除旧文件。
10. 前端必须在提交前进行基础校验。
11. 后端必须再次执行同等约束校验。
12. 任一上传文件不合法时，该批次上传必须整体失败。
13. 整批失败时，不得保存任何本批次新文件。

### 4. 样本试听与管理

1. 每个人物必须支持试听已上传样本。
2. 试听按钮默认播放最新上传的一条样本。
3. 每条样本必须支持删除。
4. 每个人物必须支持重命名。
5. 删除样本后，人物样本统计必须同步更新。
6. 当删除后人物不再满足就绪条件时，其状态必须切换为“未就绪”。

### 5. 人物就绪规则

1. 人物满足以下条件后，才视为“已就绪”：
   - 至少 `3` 段录音
   - 总录音时长至少 `30` 秒
2. 未满足条件的人物必须标记为“未就绪”。
3. 未就绪人物不得在聊天页中被选择。
4. 系统应在人物管理页明确展示未就绪原因或进度信息。

### 6. 人物设定编辑

1. 系统必须支持编辑人物设定字段。
2. 设定字段允许为空。
3. 设定变更后，后续回复必须体现新的设定。
4. 当人物设定与历史对话冲突时，后续回复必须以当前人物设定为准。

### 7. 聊天输入

1. 聊天页必须支持文字输入。
2. 聊天页必须支持语音输入。
3. 语音输入必须先经过语音识别。
4. 语音识别成功后，系统必须展示识别文本。
5. 语音识别失败时，系统必须提示失败。
6. 语音识别失败时，用户必须可以编辑文本后手动发送。
7. 聊天请求发起时，必须明确包含当前选择的人物。

### 8. 回复生成与语音输出

1. 后端必须基于以下信息生成回复：
   - 当前人物设定
   - 当前人物共享上下文
   - 用户当前输入内容
2. 回复必须避免出现“作为 AI”之类免责声明式表述。
3. 回复输出必须包含可展示的文本内容。
4. 回复输出必须包含可播放的语音结果。
5. 当目标人物音色生成不可用或效果不稳定时，系统必须回退到默认 TTS 声音。
6. 即使回退到默认 TTS，回复文本仍必须继续使用该人物设定和共享上下文生成。
7. 系统允许在回复过程中展示处理中状态。
8. 处理中状态至少应支持以下阶段表达：
   - 识别中
   - 思考中
   - 合成中

### 9. 上下文与聊天记录

1. 上下文必须按人物共享，而不是按页面会话隔离。
2. 每个人物必须保留最近 `20` 轮上下文。
3. `1` 轮定义为：用户一条消息 + AI 一条消息。
4. 每个人物必须独立保留最近 `10` 轮聊天记录。
5. 聊天记录至少应包含：
   - 发言方
   - 文本内容
   - 语音播放信息
   - 时间戳
   - 异常状态（若有）
6. 清除上下文后，系统不得继续引用旧对话。
7. 清除上下文后，当前人物设定必须恢复为空默认值。

### 10. 历史语音与播放

1. AI 回复语音必须支持重复播放。
2. 历史消息在保留窗口内必须可再次播放。
3. 超出保留窗口的较老消息可以仅保留文本，不再保留语音文件。
4. 回复语音允许保存为本地静态资源或受控文件资源，以支持重复播放。

### 11. 删除与数据清理

1. 系统必须支持删除人物。
2. 删除人物前，前端必须进行二次确认。
3. 删除人物时，系统必须同步删除：
   - 原始声音样本
   - 人物元数据
   - 聊天记录
   - 生成语音
   - 上下文数据
4. 删除不存在的人物或资源时，系统必须返回明确错误。

## 存储与实现约束

### 1. 首版存储方案

1. 首版默认采用“文件存磁盘 + 元数据存 JSON”。
2. 原始声音样本必须持久化保存，以支持未来多次训练或再次生成。
3. 人物元数据不得只靠目录名隐式表达。
4. 聊天记录与人物元数据必须有明确的数据边界。

### 2. 可扩展性要求

1. 首版无需接入数据库。
2. 首版无需接入对象存储。
3. 代码结构必须允许未来将：
   - 文件迁移到对象存储
   - 元数据迁移到数据库
4. 前端不得依赖本地磁盘结构推导业务状态。
5. 前端只能依赖后端返回的稳定业务结构。

## 非功能性需求

### 1. 架构与边界

1. 后端必须统一封装模型能力调用。
2. 前端不得直接调用第三方 ASR、LLM、TTS 或音色克隆 API。
3. 路由层只负责 HTTP 协议适配，不承担音频转码、存储编排和业务判断细节。
4. 服务层必须显式区分：
   - 人物管理
   - 声音样本管理
   - 聊天上下文管理
   - 对话编排
   - 语音输出

### 2. 错误处理

1. 所有失败场景必须返回显式错误，不允许依赖隐式失败路径。
2. 错误响应必须区分用户可修复错误与系统内部错误。
3. 以下场景必须有稳定错误码或等价稳定标识：
   - 人物名为空
   - 人物名非法
   - 人物名重复
   - 音频格式不支持
   - 音频时长过短
   - 音频时长过长
   - 人物样本数超限
   - 人物未就绪
   - 语音识别失败
   - 回复生成失败
   - 语音合成失败
   - 文件写入失败
   - 删除目标不存在
4. 批量上传或批量写入失败时，系统必须回滚已完成的部分写入。
5. 错误文案风格应偏拟人化，便于前端直接展示。

### 3. 一致性与可维护性

1. 人物状态、样本统计、聊天记录和上下文更新规则必须稳定且可预测。
2. 聊天记录和保留窗口规则必须稳定，不得因页面刷新产生隐式重置。
3. 代码实现必须优先复用 FastAPI 和 Vue 3 官方推荐能力，避免引入无必要状态管理或复杂基础设施。

## 验收标准

### A. 人物创建与命名

1. 输入合法且未重复的人物名时，可以成功创建人物。
2. 人物名为空时，创建失败并提示必填。
3. 人物名超过 `30` 个字符时，创建失败并提示超长。
4. 人物名包含非法字符时，创建失败并提示格式错误。
5. 大小写仅不同的人物名不得重复创建。

### B. 样本上传与录音

1. 可以为人物录制一段合法音频并成功保存。
2. 可以一次为人物上传多段合法音频并成功保存。
3. 任一文件格式不支持时，整批上传失败且不产生新文件。
4. 任一文件时长小于 `3` 秒时，整批上传失败。
5. 任一文件时长大于 `60` 秒时，整批上传失败。
6. 当人物已有 `50` 个样本时，不允许继续上传新样本。
7. 服务端保存后的样本格式统一为 `wav`。

### C. 人物就绪状态

1. 样本少于 `3` 段时，人物显示为“未就绪”。
2. 样本数满足但总时长不足 `30` 秒时，人物显示为“未就绪”。
3. 同时满足样本数和总时长后，人物显示为“已就绪”。
4. 未就绪人物在聊天页不可选择。

### D. 人物设定

1. 可以保存人物设定字段。
2. 更新人物设定后，后续回复可体现设定变化。
3. 清除上下文后，人物设定恢复为空默认值。

### E. 聊天流程

1. 选择已就绪人物后，可以通过文字发送消息并获得语音回复。
2. 语音回复同时展示对应文本。
3. 选择已就绪人物后，可以通过语音输入发送消息并获得语音回复。
4. 语音输入后可展示识别文本。
5. 语音识别失败时，用户可编辑文本并继续发送。
6. 聊天页可展示处理中的阶段状态。

### F. 上下文与历史

1. 同一人物连续聊天时，后续回复可引用前面内容。
2. 每个人物仅保留最近 `20` 轮上下文。
3. 每个人物仅保留最近 `10` 轮聊天记录。
4. 清除上下文后，后续回复不再引用旧对话。

### G. 回退与异常

1. 当目标人物音色不可用时，系统回退到默认 TTS 声音。
2. 回退后仍使用该人物设定生成文本回复。
3. 语音识别失败、回复生成失败、语音合成失败时，页面均有明确错误提示。

### H. 删除

1. 删除人物后，其原始样本、人物元数据、聊天记录、生成语音和上下文全部移除。
2. 删除不存在的人物时，返回明确错误。

## 备注

本规格只定义业务行为、边界和验收标准，不在本文件中展开技术任务拆解。后续若进入实现阶段，再补充对应的 `tasks.md` 和 `test-plan.md`。
