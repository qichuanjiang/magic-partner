# 图片上传并保存

## 功能概述

提供一个面向素材收集场景的图片上传与管理工具。用户输入图片简称后，可一次上传最多 10 张图片，系统将图片持久化保存到项目静态资源目录，并提供按简称分组的浏览、预览和删除能力。当前默认使用本地文件系统存储，后续可平滑替换为云端 OSS。

## 宪法对齐检查

- 是否以规格先行：是。本文件定义了上传、列表、预览、覆盖、删除及未来存储替换的完整边界，作为实现依据。
- 是否遵循最小必要功能：是。首版仅覆盖素材上传保存、列表查看、缩略图预览、删除，不包含登录、搜索、数据库元信息管理。
- 是否显式处理错误：是。文件类型、大小、简称非法、批量部分非法、重名覆盖确认、删除目标不存在等情况均要求显式返回错误。
- 是否避免不必要抽象：部分保留扩展点。首版默认本地文件系统实现，但要求定义可替换的存储接口边界，以支持后续迁移到 OSS。
- 是否满足先测后实现：是。验收标准要求先编写失败测试，再实现后端接口与前端交互。

## 功能性需求

### 1. 上传入口

1. 系统必须提供图片上传页面，允许用户输入一个“图片简称”并选择图片文件。
2. 一次上传任务只能对应一个图片简称。
3. 一次上传任务最多允许上传 10 张图片。
4. 页面必须在上传前展示当前支持的文件格式和大小限制。
5. 页面必须在用户选择图片后提供待上传文件列表。

### 2. 图片简称规则

1. 图片简称为必填项。
2. 图片简称不得为空字符串或仅包含空格。
3. 图片简称仅允许中文、英文、数字、中划线 `-`、下划线 `_`。
4. 若图片简称不合法，系统必须阻止上传，并明确提示用户错误原因。
5. 图片简称作为存储目录名使用，首版目录规则固定为 `frontend/public/image/<简称>/`。

### 3. 文件校验规则

1. 单张图片支持格式仅限 `jpg`、`jpeg`、`png`、`webp`。
2. 单张图片大小不得超过 5MB。
3. 若上传批次中任一文件不满足格式或大小要求，则整批上传失败。
4. 整批失败时，系统不得保存该批次中的任何文件。
5. 上传失败时，系统必须返回导致失败的明确错误信息。

### 4. 存储与命名规则

1. 系统必须将图片保存到 `frontend/public/image/<简称>/` 目录。
2. 若目标简称目录不存在，系统必须自动创建该目录。
3. 若同一批上传中的多个文件原始文件名相同，系统必须自动重命名，规则为 `a.png`、`a(1).png`、`a(2).png`。
4. 若目标目录中已存在同名文件，系统不得直接覆盖。
5. 系统必须先返回“存在同名文件待覆盖”的结果，由前端统一发起一次性确认。
6. 用户确认后，系统才可对本批次中所有冲突文件执行覆盖。
7. 用户取消覆盖后，系统不得写入任何文件。

### 5. 上传结果与页面流转

1. 上传成功后，系统必须返回成功状态。
2. 上传成功后，页面必须提示“保存成功”。
3. 上传成功后，页面必须自动跳转或切换到对应简称目录的图片列表视图，便于用户立即核对结果。
4. 上传成功响应中应至少包含简称、保存成功的文件列表，以及用于后续查看该目录的标识信息。

### 6. 文件夹列表

1. 系统必须提供“查看所有已保存图片”的入口。
2. 首层列表必须按图片简称展示文件夹，而不是直接平铺全部图片。
3. 每个文件夹列表项必须展示：
   - 图片简称
   - 图片数量
   - 最近更新时间
   - 一张封面缩略图
4. 文件夹列表必须按最近更新时间倒序排列。
5. 若当前没有任何已保存图片，系统必须展示空状态提示。

### 7. 文件夹内图片列表

1. 用户进入某个简称目录后，系统必须展示该目录下的所有图片。
2. 每张图片必须提供缩略图预览。
3. 图片列表必须按文件修改时间倒序排列。
4. 每张图片应展示至少以下信息：
   - 文件名
   - 最近修改时间
   - 预览图
5. 前端不得直接拼接本地静态目录路径来实现管理能力，图片读取和列表数据必须通过后端接口提供。

### 8. 删除能力

1. 系统必须支持删除单张图片。
2. 系统必须支持删除整个简称文件夹。
3. 删除单张图片前，前端必须进行二次确认。
4. 删除整个简称文件夹前，前端必须进行二次确认。
5. 删除行为为物理删除，不做软删除。
6. 当某个简称目录中的最后一张图片被删除后，系统必须自动删除空目录。
7. 若删除目标不存在，系统必须返回明确错误，不得静默成功。

### 9. 接口职责

1. 后端必须提供图片上传接口。
2. 后端必须提供简称文件夹列表接口。
3. 后端必须提供单个简称目录下图片列表接口。
4. 后端必须提供图片访问或预览所需的信息接口。
5. 后端必须提供删除单张图片接口。
6. 后端必须提供删除整个简称文件夹接口。
7. 图片相关操作必须统一经由后端完成，避免前端直接依赖本地目录实现业务逻辑。

### 10. 可扩展性预留

1. 首版无需接入数据库存储图片元信息。
2. 首版代码应预留读取和组装元信息的扩展点，便于后续接入数据库。
3. 首版存储实现默认使用本地文件系统。
4. 存储层设计必须支持未来替换为 OSS，而不要求前端交互协议发生破坏性变化。

## 非功能性需求

### 1. 架构解耦

1. 存储逻辑不得散落在路由层中，必须封装为可替换的存储服务边界。
2. 当前本地文件系统实现与未来 OSS 实现必须共享统一的领域行为定义，例如：
   - 保存文件
   - 检查冲突
   - 列出简称目录
   - 列出目录下图片
   - 删除文件
   - 删除目录
3. 前端不得依赖具体文件系统路径规则来推导业务状态，必须以接口返回结果为准。
4. 后端返回的数据结构应保持稳定，避免未来切换存储后影响前端页面结构。

### 2. 错误处理

1. 所有失败场景必须返回显式错误，不允许依赖隐式失败路径。
2. 错误响应必须区分用户可修复错误与系统内部错误。
3. 以下场景必须有明确错误码或等价的稳定错误标识：
   - 图片简称为空或非法
   - 文件格式不支持
   - 文件大小超限
   - 单次上传文件数超过 10 张
   - 批量上传中存在非法文件导致整批失败
   - 存在同名文件需要确认覆盖
   - 删除目标不存在
   - 文件系统写入失败
   - 文件系统删除失败
4. 当文件系统操作部分成功且整体应失败时，系统必须回滚已写入文件，保证批次一致性。
5. 错误提示文案必须便于前端直接展示给用户。

### 3. 一致性与可维护性

1. 上传、覆盖、删除等批量操作必须具有一致的事务语义，避免出现部分成功、部分失败但页面无从判断的状态。
2. 文件夹列表和图片列表的排序规则必须稳定且可预测。
3. 缩略图预览必须基于可直接访问的图片资源或受控的读取接口，不得要求用户手动拼接 URL。
4. 代码实现必须优先复用 FastAPI 和 Vue 3 官方推荐能力，避免引入无必要状态管理或抽象层。

## 验收标准

### A. 上传成功

1. 输入合法简称，上传 1 张合法图片，返回成功并写入 `frontend/public/image/<简称>/`。
2. 输入合法简称，上传多张合法图片，返回成功并全部写入目标目录。
3. 上传成功后，文件夹列表中显示新简称目录、正确的图片数量、最近更新时间和封面缩略图。
4. 上传成功后，页面自动进入对应简称目录，并展示刚上传的图片缩略图。

### B. 简称校验

1. 简称为空时，上传失败并提示必填。
2. 简称仅包含空格时，上传失败并提示不能为空。
3. 简称包含非法字符时，上传失败并提示格式错误。

### C. 文件校验

1. 上传单张非图片文件时，整批失败，目标目录不产生任何新文件。
2. 上传单张超过 5MB 的图片时，整批失败，目标目录不产生任何新文件。
3. 一次上传超过 10 张图片时，整批失败并提示超出数量限制。
4. 批量上传中混入 1 张非法文件时，整批失败，合法文件也不得落盘。

### D. 重名与覆盖

1. 同一批上传内出现两个同名文件时，系统自动重命名保存，文件均可在目录中看到。
2. 目标目录中已存在同名文件时，系统返回需要覆盖确认的结果。
3. 用户取消覆盖时，不产生任何写入。
4. 用户确认覆盖后，目标文件被覆盖，非冲突文件正常保存。

### E. 列表与预览

1. 文件夹列表默认按最近更新时间倒序。
2. 文件夹详情中的图片列表默认按文件修改时间倒序。
3. 文件夹列表为空时，展示空状态。
4. 进入文件夹后，可看到每张图片的缩略图、文件名和最近修改时间。

### F. 删除

1. 删除单张图片后，该图片从列表中消失，磁盘文件被实际移除。
2. 删除某目录中的最后一张图片后，该目录自动删除。
3. 删除整个简称目录后，该目录及其图片全部移除。
4. 删除不存在的图片或目录时，返回明确错误。
5. 删除操作均需要前端确认后才执行。

### G. 可替换存储

1. 上传、列表、删除接口不得暴露仅适用于本地文件系统的内部实现细节。
2. 本地存储实现可在不改动前端页面交互的前提下，被后续 OSS 实现替换。
